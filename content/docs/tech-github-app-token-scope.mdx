## GitHub App Tokenのスコープ

`actions/create-github-app-token` で発行されるトークンのスコープは、`owner` / `repositories` / `permission-*` の指定の組み合わせで決まる。直感に反する挙動があるので整理しておく。

### ownerのみ指定するとinstallation全体スコープになる

`owner` だけを指定すると、そのGitHub App installationがアクセスできる全リポジトリへの権限を持つトークンが発行される。`repositories` も `permission-*` も未指定なら、インストール時に設定された権限がそのまま使われる。READMEにも次のように書かれている。

> If `owner` is set and `repositories` is empty, access will be scoped to all repositories in the provided repository owner's installation.

```yaml
- uses: actions/create-github-app-token@v2
  id: app-token
  with:
    app-id: ${{ secrets.APP_ID }}
    private-key: ${{ secrets.APP_PRIVATE_KEY }}
    owner: flier-inc
```

### ownerを省略すると実行元リポジトリのみになる

`owner` を省略した場合、デフォルトでワークフロー実行元リポジトリ（context.repo）だけにスコープが絞られる。複数リポジトリを操作したいワークフローではここが罠になる。

> If `owner` and `repositories` are empty, access will be scoped to only the current repository.

```yaml
- uses: actions/create-github-app-token@v2
  id: app-token
  with:
    app-id: ${{ secrets.APP_ID }}
    private-key: ${{ secrets.APP_PRIVATE_KEY }}
    # ownerを指定しない場合は実行元リポジトリのみ
```

### permission-*は指定した瞬間にallowlist方式に切り替わる

`permission-*` は「all-or-nothing」ではない。正確には、一つでも `permission-*` を指定すると、指定していない権限は明示的に none として扱われる。

- 一つも指定しなければ、GitHub App自体に設定された権限がそのまま全部使われる
- 一つでも指定すると、指定した権限だけが有効になり、残りは自動的に除外される。App側でその権限を許可していてもトークンには含まれない

```yaml
- uses: actions/create-github-app-token@v2
  id: app-token
  with:
    app-id: ${{ secrets.APP_ID }}
    private-key: ${{ secrets.APP_PRIVATE_KEY }}
    owner: flier-inc
    permission-contents: read
    # contents:read のみ有効。issuesやpull-requestsなど
    # App側で許可されている他の権限はすべてnoneになる
```

「必要な権限を1つ追加しただけ」のつもりで `permission-*` を書くと、それまで暗黙に使えていた他の権限が全部落ちる。最小権限化には便利だが、指定するなら必要な権限を漏れなく列挙すること。

この挙動はアクションのREADMEには明記されていない。背後にあるREST API（installation access token作成）の仕様によるもので、公式ドキュメントには次のようにある。

> If permissions is not specified, the installation access token will have all of the permissions that were granted to the app.

裏を返せば、permissionsを指定した場合はそこに書いた権限だけがトークンに付与される。

### 注意

- `permission-*` を後から追加した際に、既存処理が権限不足で壊れていないか確認する
- installation全体スコープのトークンは影響範囲が広いため、必要なければ `repositories` で絞る
